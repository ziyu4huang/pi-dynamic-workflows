## 1. Executive summary

pi-dynamic-workflows already matches the structural shape of Claude Code's **Workflow tool**: the same primitives (`agent`/`parallel`/`pipeline`/`workflow`/`log`/`phase`/`budget`/`args`), the same caps (`min(16, cpu-2)` concurrent, 1000 agents/run), a real synthetic `structured_output` tool with typebox validate-and-retry on bad args, a coherent model-tier precedence, a vm-sandboxed script runtime, and — uniquely — a **disk-backed run journal** that already gives pi a resume head start over CC's in-session-only workflow resume.

The 2.0.0 thesis is two-pronged:

1. **Align** on the dimensions where pi silently diverges from CC's Workflow-tool contract. Five of these are correctness footguns, not missing features: `agentType` is a prose hint that binds nothing; structured output fails *silently to null* on weak/non-Claude models; resume is positional (stale-downstream) rather than longest-unchanged-prefix; the hard budget is both racy under fan-out and **never armed in production** (`DEFAULT_TOKEN_BUDGET = null`); and the determinism guard is a bypassable parse-time regex.

2. **Surpass** CC where pi's disk journal + deterministic replay let it do things CC's Workflow tool structurally cannot: crash-safe durable resume, journaled-and-replayable human checkpoints, persisted per-agent cost/cache/timing observability, per-phase sub-budgets, and a reusable quality-pattern library.

This is a **major (2.0.0)** bump because the resume cache semantics change (positional → longest-unchanged-prefix, plus `agentType`/phase-match feed the resume hash), so journals written by 1.x will not cache-hit identically on 2.0 — an intentional, breaking correctness change.

A recurring correction runs through this plan: the common framing benchmarks pi against **CC subagents** or **CC agent-teams**. The correct benchmark is the **CC Workflow tool**. Several "gaps" (schema+retry, quality patterns, hard budget) only look like gaps or wins under the wrong benchmark.

### P0 — Correctness & parity floor (ship together; this is what makes 2.0 a defensible major)

#### P0-1 — Real `agentType` registry: bind tools + model + system-prompt (not prose)

**Problem.** `agentType` is the single largest behavioral divergence from CC. Today it is a prose hint: one line `Act as workflow subagent type: X` appended to the prompt and folded into the resume hash, binding no tools/model/prompt. Every subagent runs identical `createCodingTools(cwd)`. It is undocumented in `workflow-tool.ts`, so authors can't even discover it.

**Design (concrete).**
- New standalone module `src/agent-registry.ts`. Scan `.pi/agents/*.md` (project) then `~/.pi/agents` (user), parsing frontmatter with the **same front-matter helper `skills` machinery uses** (`loadSkillsFromDir`/`SkillFrontmatter` in `@earendil-works/pi-coding-agent` is the template — note: there is **no** existing subagent loader to reuse; this is built from scratch). Implement **project > user** precedence with silent dedup-on-name. **Resolve the registry once per run and snapshot it**, so two `agent()` calls in the same run can't observe a mid-run edit (determinism), while a later resume re-reads and can invalidate.
- **Resolve `agentType` once, in the workflow layer** (`agent()` at `workflow.ts:206`), *not* also inside `WorkflowAgent.run` (`agent.ts:163`). The fake-agent suite injects `options.agent` (a `Pick<WorkflowAgent,'run'>`) and bypasses `WorkflowAgent` entirely; resolving in both places would be invisible to tests and could double-apply filters. Resolve in `agent()`, then pass already-resolved `{filteredTools, modelSpec, instructionsBody}` into `run()` via existing options.
- **Tool binding by FILTERING the `ToolDefinition[]` by name in the workflow/agent layer** — *not* by forwarding SDK `excludeTools`/`noTools` strings. Those SDK options operate on built-in tool name strings + extension tools; pi's coding tools are passed as `customTools` **objects** from `createCodingTools(cwd)`, and subagents run on an in-memory session with no extension discovery, so the SDK string-allowlist will not bind them. **Documented precedence (we do not claim CC-exactness — the SDK docs the opposite order):** start from the agent's `tools` allowlist (or all coding tools if omitted), then remove `disallowedTools`.
- **Model:** feed the definition's model into `resolveAgentModelSpec` (`agent.ts:28`) as a level between tier and phase. Final order: explicit `opts.model` > `agentType.model` > tier > phase > (P2-1's) `meta.model` > session default.
- **System prompt (be honest):** for v1, **prepend the markdown body via the existing instructions mechanism** (`buildPrompt()` injects a user-message prefix at `agent.ts:253`). A *true* system prompt would require a per-subagent `DefaultResourceLoader` with `appendSystemPrompt` passed as `session.resourceLoader` — noted as a follow-up, not v1.
- **Resume:** hash the **resolved** definition (tools+model+prompt) into `hashAgentCall` (`workflow.ts:612-627`) using the per-run snapshot, so editing an agent `.md` invalidates cached results on a later resume without intra-run nondeterminism.
- **Discoverability:** surface available `agentType`s in `workflow-tool.ts` `promptGuidelines`, mirroring `modelRoutingGuideline()`/`listAvailableModelSpecs()`.
- **Scope which keys bind vs. are parsed-but-ignored:** **bind** `tools`, `disallowedTools`, `model`, `prompt`; **parse-but-ignore** (documented) `mcp`, `background`, `skills`, `isolation` until each is independently wired.

**Why it aligns / surpasses.** Closes the single largest CC-divergence and matches Anthropic's #1 reliability lever (role-bound task definitions with objective/output-format/tools/boundaries). Files are *near*-portable with CC (`.pi/agents` vs `.claude/agents`) — we do not overclaim full portability.

**Best practices.** Detailed bound task definitions; registry precedence; resolved-definition hashing for resume correctness.

**Effort / risk.** **L / medium** — touches the hot `agent()` path and resume hashing; the whole parser/precedence/dedup/filter is net-new. Mitigated by single-point resolution and the fake-agent suite, then a real-pi run.

**Real-pi test plan.** Create `.pi/agents/security-auditor.md` (frontmatter: `tools: [read, grep]`, `disallowedTools: [write, bash]`, `model: <small-tier>`, body = a security-reviewer prompt). Run a real pi workflow whose script calls `agent("audit this dir", { agentType: "security-auditor" })`. Verify in `/workflows` that the agent (a) ran on the small-tier model, (b) had write/bash absent from its tool list, (c) used the markdown body as guidance. Then edit the `.md` model line and **resume** the same run; confirm that agent's cached result is invalidated and re-runs, while unrelated calls cache-hit.

#### P0-2 — Schema forcing-by-restriction + bounded no-call retry (works on weak/non-Claude models)

**Problem.** pi has genuine bad-args retry, but the "model emitted prose and stopped without calling `structured_output`" miss class has **zero** retry: `capture.called` stays false, `agent.ts` throws recoverable, `agent()` returns **null silently** (`workflow.ts:322-323`). Weak/non-Claude small-tier models — exactly the routes pi encourages — hit this most.

**Design (concrete). Provider `toolChoice` forcing is NOT used — it is infeasible here.** `toolChoice` exists only on raw per-provider option types consumed by low-level stream functions; it is absent from `StreamOptions`, `SimpleStreamOptions`, the harness-curated `AgentHarnessStreamOptions`, `CreateAgentSessionOptions`, and `PromptOptions`. There is no way to pass it through the API `agent.ts` uses without forking the SDK. We drop that layer entirely.

- **(A) Forcing-by-restriction + bounded repair.** After `session.prompt()` returns with `capture.called === false`: (1) optionally `session.setActiveToolsByName(['structured_output'])` (takes effect next turn) to remove other tools; (2) re-prompt up to `maxSchemaRetries` (default 2) with `"You did not call structured_output. Call it now as your only action."` `capture` persists across prompts in the same session, so a later turn that finally calls the tool populates `capture.value` and is validated/retried at the tool layer as today. **Each repair turn must:** check `options.signal` (throw if aborted), wrap in the existing `withTimeout`, and route its tokens through `onUsage`/`recordTokens` so the hard budget is honored. Tool-set restriction alone does **not** force a call — the real forcing mechanism on pi *is* the bounded re-prompt.
- **(B) Strict, gated, last-resort extraction.** Only if repair still fails **and** the final prose contains a clearly delimited JSON block: parse it, run it through the same typebox `Compile + Value.Convert/Clean/Decode` against the caller's `TSchema`, and on success **log loudly** that the result came from prose extraction, not a tool call. On any failure, **do not fabricate**. (Note: `structured-output.ts` today relies on pi's built-in tool-arg validation, so this in-code validation path is small *new* code that must mirror pi's semantics.) A coerced-but-wrong object that passes typebox is worse than a logged null — extraction stays strict last-resort, never routine.
- **(C) Distinct non-recoverable code.** Add `WorkflowErrorCode.SCHEMA_NONCOMPLIANCE`. When repair (+optional extraction) is exhausted, throw `WorkflowError(msg, SCHEMA_NONCOMPLIANCE, { recoverable: false, agentLabel })`. `wrapError` passes `WorkflowError` through, so `workflow.ts:325` re-throws instead of silent-null, and `logger.error` already fires at `workflow.ts:317`. Honors "no silent caps."

**Why it aligns / surpasses.** Reaches parity with CC's Workflow tool (which forces+retries `StructuredOutput`) on Claude, and **surpasses** specifically on the non-Claude/small-tier routes pi uniquely encourages — exactly the stated problem.

**Best practices.** Repair + rules-first coercion before declaring failure; surface the give-up loudly.

**Effort / risk.** **M / medium** — repair turns cost tokens (must hit the budget) and the extraction validator must mirror tool-layer semantics. No SDK fork.

**Real-pi test plan.** Run a real pi workflow with a schema-bearing `agent()` routed to a known-weak small-tier/non-Claude model on a prompt that historically yields prose. Confirm: (1) the repair re-prompt recovers a validated object (visible in `/workflows`); (2) when forced to fail (e.g. impossible schema), the run **halts with `SCHEMA_NONCOMPLIANCE`** and a logged line, not a silent null. Fake-agent unit tests: a fake session returning prose-then-tool-call (proves repair recovers) and prose-forever (proves the error surfaces).

#### P0-3 — Longest-unchanged-prefix resume (replace positional cache)

**Problem.** Resume is positional: `workflow.ts:244` serves any call whose `index+hash` match, with no suffix invalidation. CC's contract: the first edited/new call **and everything after** runs live. pi serves stale downstream results when an upstream edit should logically invalidate later same-text calls; an index shift can false-hit an adjacent identical sibling.

**Design (concrete).** Add `firstMiss: number` to `RuntimeState` (init `Infinity`). `callIndex` is assigned synchronously at `workflow.ts:239` *before* the limiter and *before* the cache check at `:244`, so prefix order is stable across runs for lexical + `parallel()` cases. Compute `serve = cached && cached.hash === callHash && callIndex < state.firstMiss`; if not `serve`, set `state.firstMiss = Math.min(state.firstMiss, callIndex)` before the limiter block, then run live. Single comparison on the hot path.

**Honest scoping (do not claim CC-exactness universally).** This gives CC's exact "cached prefix then everything-after-live" for **straight-line and `parallel()` fan-out only**. In `pipeline()`, stage-2+ `agent()` calls are entered in item-**completion** order after awaits, so their `callSeq` is genuinely nondeterministic across runs — a pre-existing limitation `firstMiss` neither fixes nor regresses. Document it; track pipeline determinism separately. Nested `workflow()` runs get `resumeJournal: undefined` and always run fully live — `firstMiss` has no effect there (consistent with current behavior).

**Why it aligns.** Matches CC's documented resume contract for the deterministic cases; eliminates a correctness footgun. Not a surpass.

**Effort / risk.** **S / low** — localized to the cache gate.

**Real-pi test plan.** Real pi run of a 3+ call straight-line workflow. Edit **call index 1** and resume; assert in `/workflows` that an **unedited call index 2 re-runs live** (live calls = 2, not 1) — this is the suffix-invalidation contract the existing tests do *not* cover (the current "hash mismatch" test only edits the last call). Add a `parallel()` case: edit one thunk in a fan-out of identical prompts; assert every same-or-later index re-runs.

#### P0-4 — Arm the hard budget, enforce it under fan-out, stop swallowing non-recoverable throws

**Problem (three layers, biggest first).**
1. **The budget is never armed in production.** `DEFAULT_TOKEN_BUDGET = null` (`config.ts:15`); `WorkflowManager.runWorkflow` passes no `tokenBudget`; the gate `if (budget.total !== null …)` never fires in real runs. CC's distinctive edge **does not exist on pi today.** Polishing the gate's concurrency accounting without arming the budget is optimizing dead code.
2. Non-recoverable throws (`TOKEN_BUDGET_EXHAUSTED`, `AGENT_LIMIT_EXCEEDED`) thrown *before* the limiter are **swallowed to null** inside `parallel()` (`workflow.ts:344-347`) and `pipeline()` (`:371-374`) catches that `wrapError` then unconditionally `return null`, never checking `.recoverable`. So the hard stop leaks under fan-out.
3. `shared.spent` is incremented only in `recordTokens` *after* an agent completes, so N concurrent agents at the gate all observe `spent=0`.

**Design (concrete).**
- **(1) Wire a real budget end-to-end.** Thread a configurable `tokenBudget` from a user setting / default through `WorkflowManager.runWorkflow` and the workflow tool into `runWorkflow`. Without this, (2)/(3) operate on a gate that never arms.
- **(2) Re-throw non-recoverable in fan-out.** In both catches, after `const workflowError = wrapError(error)`, add `if (!workflowError.recoverable) throw workflowError;` before the log/return-null — so exhaustion halts the run inside fan-out exactly as a directly-awaited `agent()` does at `:325`. **This is the highest-value, lowest-risk piece.**
- **(3) Match CC's soft-gate semantics correctly (NOT a stricter ceiling).** **Drop optimistic pre-reservation/true-up** — it needs a pre-run per-agent token estimate that doesn't exist (`estimateTokens` needs the result), risks false `TOKEN_BUDGET_EXHAUSTED` halts or a wrong total. Per ground truth, **CC's budget is also a soft gate**: once `spent>=total`, *further* `agent()` calls throw; in-flight agents complete. Match that by making the budget check + `shared.agentCount++` an atomic synchronous critical section at the gate, so the check observes a consistent `spent` and the agent counter serializes. Accept that in-flight agents may push `spent` slightly past `total` on the same tick — that is CC's documented behavior, not a defect.

**Why it aligns.** Turns CC's "watch /workflows" prose into an enforced, *armed* guarantee; matches (not exceeds) CC's soft-gate semantics. `closesGapOrSurpasses` = closes the gap.

**Effort / risk.** **M → effective low** — removing reservation estimation eliminates the wrong-total risk.

**Real-pi test plan.** Real pi run with a deliberately small `tokenBudget` and a fan-out (`parallel()` of many real agents). Confirm: (1) the budget is actually enforced (run halts), and (2) it halts **inside** the fan-out with `TOKEN_BUDGET_EXHAUSTED` surfaced in `/workflows`, not a silent batch of nulls. Fake-agent tests: assert exhausted budget halts inside both `parallel` and `pipeline`.

#### P0-5 — Runtime determinism neutering of `Date`/`Math.random` **and the host-`Function` escape**

**Problem.** The guard is a parse-time regex (`workflow.ts:138`) protecting only 3 literal forms. Verified bypasses against the real vm context shape: `Date()`, `Date['now']()`, `const D=Date;D.now()`, `Math['random']()` all produce live nondeterministic values because `Math` is injected and `Date` is reachable. **Worse — verified — the host-`Function` escape is wide open:** injected host-realm globals (`Object`, `Array`, `Promise`, `Set`, `Map`) expose host `Function` via `.constructor`, so `Object.constructor('return Date.now()')()` returns a live timestamp and `Promise.constructor('return globalThis.process')()` reaches the host `process` — a sandbox escape beyond mere determinism. Any bypass breaks resume's same-script→100%-cache-hit guarantee.

**Design (concrete).** Neuter at the runtime layer in the vm context construction (keep the regex as a fast author hint).
- **Math:** `Object.create(Math)` then `Object.defineProperty(ctxMath, 'random', { value: () => { throw … } })`. **Do NOT spread `{...Math}`** — Math's properties are non-enumerable, so `{...Math}` is `{}` and drops every method.
- **Date wrapper:** `function DDate(...a){ if (a.length===0) throw …; return Reflect.construct(RealDate, a, DDate); }`; keep deterministic statics (`UTC`, `parse`), make `now` throw; set `DDate.prototype = Object.create(RealDate.prototype)` with `constructor` redefined to `DDate` so `new Date(arg).constructor.now()` can't leak the real `Date`. Allow `new Date(arg)` with explicit args (matches CC).
- **CRITICAL — close the host-`Function` escape** (the proposal's "closes all vectors" claim was false without this): either (a) construct the injected globals *inside* the vm via an in-context bootstrap so their `.constructor` is the in-context `Function`, then neuter that `Function`; or (b) prefer running scripts with vm's own globals and inject **only the workflow API surface** rather than host `Object`/`Array`/etc. Recommend (b) where feasible — smallest attack surface.
- **Optional surpass (deterministic builtins).** Provide a run-pinned clock and a seeded RNG, recording `seed`+`epoch` as **new top-level fields on `PersistedRunState`** (`run-persistence.ts` has only a per-index journal today) so resume reproduces them. This — not the neutering — is the part that surpasses CC.

**Why it aligns / surpasses.** Runtime enforcement is the correct boundary (CC also enforces at runtime → parity on mechanism). The seeded-clock/RNG genuinely exceeds CC, which only throws.

**Best practices.** Enforce invariants at the runtime layer, not by lexical pattern-matching.

**Effort / risk.** **M / low-medium** — corrected from the proposal's "S": the host-`Function` fix and the persisted seed channel are real work.

**Real-pi test plan.** Real pi workflow whose script attempts each vector — `Date()`, `Date['now']()`, `const D=Date;D.now()`, `Math['random']()`, `Object.constructor('return Date.now()')()`, `Set.constructor('return Date')()` — and asserts each **throws** at runtime; plus positive cases that `new Date(0)`, `Math.max`, `Math.floor` still work. If the seeded surpass ships: run with the RNG, resume, and confirm identical values.

### P1 — Durability, observability, and quality library (the "surpass" tier)

---

#### P1-1 — Crash-safe durable disk resume (Phase A only for 2.0; defer the daemon)

**Problem.** pi persists journals but: whole run JSON is rewritten on every agent completion with **no atomic write** (mid-write crash corrupts it); `load()` swallows the parse error and loses everything; no crash-recovery sweep (stale `running` rows survive); no lease (two processes can resume the same `runId`).

**Design (concrete) — PHASE A only, shipped as 2.0.**
1. **Atomic persistence** in `run-persistence.ts` `save()`: write `${runId}.json.tmp`, then `renameSync` over the target (atomic on the same fs). **Keep the single-JSON layout** — do **not** introduce the `agent-<id>.jsonl` sidecar; that name is CC's layout lifted from ground truth, and pi's embedded journal already round-trips.
2. **Recoverable load:** on each successful save also write `${runId}.json.bak`; on `JSON.parse` failure attempt the `.bak` and surface a structured warning instead of returning null blindly.
3. **Crash-recovery sweep** on `WorkflowManager` construction: for any persisted `status==='running'` not in the in-memory map, reconcile to **`paused`** (never `failed` — preserves the journal so `resume()` works).
4. **Lease via `O_EXCL` lockfile, not mtime:** on `resume()`/`startInBackground`, create `${runId}.lock` with `wx` flag containing `{pid, startTime}`, keyed on the **absolute run-file path** (runIds are timestamp+random and can collide across projects). If it exists and the pid is dead (`process.kill(pid,0)`), steal it; if alive, refuse with "already running in PID n". Release on completion/abort. (mtime leases are unreliable across filesystems.)

**Phase B (DEFERRED, not 2.0).** A per-user daemon that *owns running runs across CLI exit* is a net-new standalone Node subsystem: the SDK exposes **no** daemon/supervisor/IPC primitive (only `RpcClient`/`runRpcMode` for the CLI's own RPC). It also has an unresolved scoping mismatch — runs are per-**project** (`.pi/workflows/runs`) but the proposed daemon is per-**user**. Defer until users actually want runs to keep executing after CLI close; treat as its own release. Do **not** make "survives terminal close" the default for any background run in 2.0.

**Why it aligns / surpasses.** Phase A makes pi's *disk* resume crash-safe and concurrency-safe — strictly beyond CC's in-session-only Workflow resume. Honest changelog framing: "crash-recovery + safe-resume," not "a live daemon." (Note: agents in-flight at crash time were never journaled — `onAgentJournal` fires only on success — so on respawn they re-run live, identical to CC, not better.)

**Effort / risk.** **Phase A: M / low.** (Daemon Phase B: L / high — deferred.)

**Real-pi test plan.** Real pi run; `kill -9` the process mid-run; relaunch and confirm the run reconciles to `paused`, `resume()` replays the journaled prefix and finishes. Corrupt the primary JSON, leave `.bak` intact, confirm it loads with a warning. Attempt to `resume()` the same `runId` from a second process and confirm it's refused; kill the first, confirm the second can steal the stale lock.

#### P1-2 — Quality-pattern standard library (injected globals, not imports)

**Problem.** pi ships the patterns only as one-off generated templates; authors must re-derive them by reading source. They are not composable primitives. (Correction: the patterns are **par** with CC, which already *recommends* them — this is an ergonomics win, not a capability surpass.)

**Design (concrete). Delivered as INJECTED GLOBALS on the `vm.createContext` object** alongside `agent`/`parallel`/`pipeline` (`workflow.ts:411-441`) — **not** imports. The vm has no module loader and `import`/`require`/`fs` are forbidden. Advertise each in `workflow-tool.ts` `promptGuidelines` so the model actually uses them.

Phase 1 (high value, low risk), built purely on `agent()`/`parallel()` so `callSeq` ordering stays deterministic and resume keeps working:
- `verify(item, {reviewers=2, threshold=0.5, lens?})` — N skeptics forced to refute via schema `{real:boolean, reason}`, default `real=false`.
- `judgePanel(attempts, {judges, rubric})` — each judge is **one** schema-enforced call `{score:0..1, pass:boolean, reason}`; the deterministic script picks max-score with **stable tie-break by input index**.
- `loopUntilDry({round, key, consecutiveEmpty=2})` — deterministic **first-seen** dedupe by stable key; **must `try/catch` `TOKEN_BUDGET_EXHAUSTED` + `AGENT_LIMIT_EXCEEDED` and return partial results**, not throw.
- `completenessCheck(args, results)` — single tail critic.

Phase 2 (optional): `verifyWithLenses`, `sweep(modes[])` under the existing cap, `ruleCheck()` shelling deterministic grep/lint/compile/test.

#### P1-4 — Per-phase sub-budgets (the real gap) + thin `retry`/`gate` helpers

**Problem.** pi has only implicit unbounded retry and a single global ceiling: no per-phase/soft sub-budget, no author-facing bounded retry, no policy gate. A noisy phase can exhaust the whole run.

**Design (concrete).**
- **(A) Per-phase / per-subtree sub-budgets — the only genuine gap vs CC's Workflow tool.** Add `phase(title, {budget?})` and `agent(..., {budget?})` carving a sub-ceiling from the run total. **Prerequisite (depends on P0-4):** today `shared.spent` is post-hoc (incremented only after an agent completes), so with concurrency up to 16 a sub-budget admission check reads stale `spent` and admits a full extra wave. The honest fix is to reuse P0-4's atomic gate accounting; accept the same one-tick overshoot CC accepts. Emit a soft-warning log at ~80% and throw `TOKEN_BUDGET_EXHAUSTED` at 100% of the sub-ceiling. Journal nothing new — sub-budget state recomputes deterministically from already-journaled per-agent results on resume.
- **(B) `retry`/`gate` as thin runtime helpers** exposed in the vm context: `retry(thunk, {attempts, until})` and `gate(thunk, validator)` that re-feeds validator feedback into the next attempt. **Drop `backoff` entirely** — no timer exists in the vm and a delay has no resume value. These are sugar over the existing for-loop-and-`agent()` pattern (the adversarial-review builtin already does this by hand); their only justification is auto-journaling each attempt under stable `callSeq` indices. **Document that retry/gate chains cache-miss-cascade on resume** (attempt N+1's hash depends on N's live result — correct, not a bug).
- **(C) Downgrade the "mid-agent budget guard" to an honest slice timeout.** "Surface `remaining()` into the running agent loop" is **not achievable** — the subagent is a separate session whose loop the workflow can only *observe and abort*. Implement opt-in abort-on-cumulative-token-threshold by subscribing to session events and calling `session.abort()`, layered on the existing `timeoutMs`/`withTimeout`. Document that it yields a **null** (recoverable), not a graceful partial.
- **(D) Drop "graceful degradation" as a primitive** — branching on `budget.remaining()` to downshift model / skip optional rounds already works today; add a `promptGuideline` + a builtin example instead.

**Why it surpasses (narrowly, and honestly).** Per ground truth, CC's Workflow tool has a hard total budget + schema-retry but **no per-phase sub-budget and no declarative retry combinator**. Sub-budgets are genuinely net-new vs CC's Workflow tool. The retry/gate/degradation parts are sugar/docs over what pi can already do.

**Effort / risk.** **M / medium** — gated on P0-4's accounting.

**Real-pi test plan.** Real pi run where `phase("noisy", {budget: small})` is exhausted by real agents; confirm the phase throws `TOKEN_BUDGET_EXHAUSTED` at its sub-ceiling while the run's overall budget is untouched and later phases proceed. Run a `gate()` whose deterministic validator (e.g. compile check) fails twice then passes; confirm re-runs with feedback and a deterministic resume.

#### P2-3 — Journaled, replayable human checkpoints (drop the inter-agent mailbox)

**Problem.** `agent()` has no user-skip/steering path; no human sign-off mid-run without sharding into separate workflows. (Correction: mid-run steering + `SendMessage` mailbox is CC's **agent-teams** primitive, not the Workflow tool — benchmarking pi's Workflow tool against agent-teams here is the wrong comparison.)

**Design (concrete).**
- **DROP the inter-agent mailbox / debate-while-running entirely.** It is fundamentally incompatible with resume: `callIndex` is assigned synchronously at lexical order *before* the limiter, and the resume hash excludes results. A live channel makes a downstream agent depend on real model-latency arrival order — the exact wall-clock nondeterminism the runtime bans. "Journal every message" does not make it exact (cached prefix replays old order while a re-run downstream agent sees different mailbox state). Adversarial debate is already deterministic **after** agents return (the `adversarial-review.ts` pattern); multi-round debate is a deterministic script loop where round k's prompts are pure functions of round k-1's journaled results.
- **KEEP a narrow deterministic checkpoint primitive.** Add a script global `checkpoint(promptText, options)` that resolves to the human's reply and is **journaled exactly like an `agent()` result** (a new `JournalEntry` kind keyed by `callSeq`, sitting in the same deterministic-prefix cache). Constraints:
  1. **Thread the `ui` context** from the tool `execute` ctx (currently ignored as `_ctx`) through `WorkflowManager` into `runWorkflow`; expose `checkpoint()` only when `hasUI`. In headless/RPC/print/detached-background mode, `checkpoint()` takes a declared default (auto-approve or auto-abort per `options`) and **journals that** — never hangs.
  2. Implement via `ui.confirm`/`select`/`input` with a mandatory timeout + `AbortSignal` wired to the run controller.
  3. Count each checkpoint against the agent-counter/abort guardrails so it can't loop forever. A human gate spends **no tokens** — gate it on the counter + abort, not budget.
  4. On resume, replay the journaled reply by `callIndex` exactly like a cached agent — **this is the genuine edge over CC**, whose steering is non-replayable and in-session only.

**Why it surpasses.** Replayable human checkpoints are something CC's Workflow tool structurally cannot do (its steering, where it exists at all, is in agent-teams and non-replayable).

**Effort / risk.** **L / high** — UI threading + detached-run default path + new journal kind. Lowest priority; build after the core lands.

**Real-pi test plan.** Two real pi runs: (1) interactive mode — script calls `checkpoint("approve plan?")`, human approves; resume the run and confirm the journaled reply replays **without re-prompting**. (2) headless/background — confirm `checkpoint()` takes the declared default and journals it rather than hanging.

## 5. Breaking changes, migration, and why this is a major (2.0.0) bump

**Breaking — resume cache semantics change (the core reason for the major bump):**
- **Positional → longest-unchanged-prefix resume** (P0-3): an upstream edit now invalidates everything after it. 1.x journals will *not* reproduce 1.x cache-hit behavior — by design.
- **`agentType` and exact phase-match now feed model/tool resolution and the resume hash** (P0-1, P2-1): calls that used to cache-hit may now re-run after upgrade if they used `agentType` or fuzzy-matched phase titles.
- **Determinism runtime neutering** (P0-5): scripts that *relied on* a `Date`/`Math.random` bypass (intentionally or not) now throw at runtime instead of parse time.

**Behavioral — previously-silent failures now surface (no silent caps):**
- Schema no-call exhaustion throws `SCHEMA_NONCOMPLIANCE` instead of returning null (P0-2).
- `TOKEN_BUDGET_EXHAUSTED`/`AGENT_LIMIT_EXCEEDED` inside `parallel`/`pipeline` now halt the run instead of becoming nulls (P0-4).
- The hard token budget can now actually be armed and enforced (P0-4) — runs that quietly ran unbounded may now stop.

**Removed surface:**
- `buildModelRoutingInstructions` deleted (P2-1); `shouldUseWorkflow`/`suggestWorkflowScript` deleted after P1-5 (P2-2) — these are public exports of a published package, so their removal is a breaking API change.
- Per-agent `run_in_background` is explicitly **not** added; whole-run backgrounding is the supported path (P2-2).

**Migration.** Persistence schema additions (cost/cache/timing in P1-3, seed/epoch in P0-5) are **additive optional fields** — old run JSON still loads. Authors who relied on stale-downstream caching, the orphaned heuristic exports, or determinism bypasses must adjust. Phase-title matching becomes case-sensitive exact — authors with overlapping fuzzy titles must rename or use the regex branch.

**Why major:** the resume contract — the central behavioral guarantee of this package — changes for the better but incompatibly, and public exports are removed. Per semver, that is a 2.0.0.

## 6. Dropped ideas (one line each)

- **Backoff in `retry()`** — no timer in the vm and no resume value (dropped from P1-4).
